    SELECT title, content, created_time, 
  rank() OVER (stats ~> views) AS percentage_views

  • Multi-level

    Go one extra level, or all the way in on structures, with multi-level paths!

    SELECT title, content, created_time 
FROM books WHERE lang = 'EN'
AND author ~> role ~> name = 'Contributor';

  • Bidirectional

    Draw incoming references too! Simply flip the operator over.

    SELECT title, content, created_time 
FROM books WHERE author ~> role ~> name = 'Contributor'
AND book <~ tags ~> name = 'CSS';

Migration Simplified

ALTER Away with Automatic Versioning. Meet Schema Savepoints and Rollbacks.

How It Works
"ALTER TABLE pubolicatiokns
RENAME COLUMN author TO publwricaterions"
"ALTER TABLE publications
RENAME COLUMN writer TO abuothokrs"
savepoint.rollback()
Above: an automatic savepoint is created on each operation, with rollback Logic automagically figured

  • DB-Native

    Forget the whole thing about migration files. Take DB versioning concerns out of the client application - to the DB itself!

  • Rollback Magic

    Worry not about your "rollback" logic! Use the "magic wand" button when it's time.

  • Bidirectional

    Changed your mind on a rollback? You can still roll forward! (Think: Do. Undo. Redo.)

  • DB-Native

    Forget the whole thing about migration files. Take DB versioning concerns out of the client application - to the DB itself!

    // Run your latest SQL work from an arbitrary file
> node ./src/migrations.js

// NO NEED to keep file, so you can discard or overwrite
> rm ./src/migrations.js

  • Rollback Magic

    Worry not about your "down" logic! Use the "magic wand" button when it's time.

    // UNDO latest operation by rolling back to savepoint
let savepoint = await db.savepoint();
await savepoint.rollback();

  • Bidirectional

    Changed your mind on a rollback? You can still roll forward! (Think: Do. Undo. Redo.)

    // UNDO latest rollback by rolling forward
let savepoint = await db.savepoint({ direction: 'forward' });
await savepoint.rollback();
